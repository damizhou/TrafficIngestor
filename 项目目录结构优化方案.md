# TrafficIngestor 项目目录结构优化方案

最后更新：2026-07-15 10:38:49

## 1. 背景与结论

项目经过长期迭代后，宿主机调度、容器内执行、浏览器差异、代理模式、数据集入口、临时工具和运行产物逐渐混合在同一层级。当前问题不只是目录数量较多，而是目录名同时承担了浏览器、网络模式、数据来源和运行实例等多种职责，导致新增场景时倾向于复制目录和入口脚本。

目录优化不应采用一次性整体搬迁。建议先分离源码、配置、输入数据、运行产物和历史资料，再收敛重复入口，最后拆分巨型基础类。迁移期间保留旧入口作为兼容包装器，确保现有 Docker 挂载、远程运行命令和结果路径不被突然破坏。

## 2. 当前主要问题

### 2.1 顶层目录缺少稳定边界

仓库根目录同时存在以下几类内容：

- 宿主机调度代码：`trafficIngestor/`、`trafficIngestor_clash/`、`multi_csv_traffic_ingestor/`。
- 容器内执行代码：`traffic_capture_single_csv*/`、`single_traffice_capture/`、`batch_traffice_capture/`。
- 公共模块：`tools/`。
- 数据处理和维护脚本：`small_tools/`。
- 输入、输出、临时文件和历史资料：`temp/`、`document/`、`monthly/`、`db/` 等。
- 第三方程序或构建内容：`clash-for-linux/`、`clash_build/`。

这些目录的生命周期和修改原因不同，却处于同一层级，难以判断哪些是稳定源码、运行工作区或可清理内容。

### 2.2 入口脚本由多个维度组合产生

现有文件名经常同时包含：

- 输入类型：单 CSV、多 CSV、数据库。
- 浏览器：Chrome、Edge、Firefox。
- 网络模式：普通、固定 IP、Clash。
- 数据集或站点：Bsky、Mastodon、Threads、X、Homepage、Subpage。

例如 `traffic_capture_single_csv_firefox_clash.py` 和 `traffic_capture_single_csv_fixed_ip_europe_clash.py` 本质上是公共采集流程加不同配置。继续使用复制脚本的方式组合这些维度，会造成入口数量持续增长。

### 2.3 基础类承担过多职责

`trafficIngestor/base_traffic_ingestor.py` 已同时负责：

- Docker 容器创建、启动、检查和删除。
- Docker 网络、固定 IP、网桥和 veth 诊断。
- 宿主机网卡 offload 配置。
- CSV 读取、修复和成功记录删除。
- 并发调度、重试、节流和进度统计。
- 产物搬运、权限修复和目录创建。
- 任务日志、结果完整性检查和清理等待。

目录调整如果不伴随职责拆分，只会把巨型文件移动到新位置，并不能改善维护成本。

### 2.4 源码目录与运行状态混合

`HOST_CODE_PATH` 既用于保存或复制 `action.py`，又可能出现 `meta/`、日志、抓包产物和运行时子目录。由此产生以下风险：

- 清理运行目录时误删源码。
- 多个采集入口共享或覆盖运行状态。
- Git 状态被运行产物污染。
- 无法明确区分可提交文件与可删除文件。

### 2.5 工具、数据和文档混合

`small_tools/` 同时包含数据转换脚本、输入数据和输出结果；`temp/` 中既可能有临时文件，也可能有仍具参考价值的设计文档。任务队列中还存在历史编码损坏内容，说明历史资料也需要明确归档和编码治理。

## 3. 优化原则

1. **按职责组织源码**：目录表达模块职责，不表达某次任务名称。
2. **配置与实现分离**：浏览器、镜像、代理、网络和数据集路径通过 profile 或运行参数组合。
3. **源码与运行状态分离**：源码目录只保存可版本控制内容，所有运行写入统一进入 `runtime/`。
4. **保持单一实现**：相同 CSV 任务流程、产物处理和 Docker 调度只保留一份实现。
5. **渐进迁移**：先增加新结构和兼容层，再迁移调用方，最后删除旧入口。
6. **先建立验证基线**：每次迁移至少执行相关文件的 `py_compile`；条件允许时再做 Docker 单站点烟测。

## 4. 建议目标结构

```text
TrafficIngestor/
├─ src/
│  └─ traffic_ingestor/
│     ├─ core/                  # 调度、任务模型、产物规则、完整性检查
│     │  ├─ scheduler.py
│     │  ├─ tasks.py
│     │  ├─ artifacts.py
│     │  └─ completeness.py
│     ├─ infrastructure/        # Docker、网络、宿主权限等外部系统适配
│     │  ├─ docker.py
│     │  ├─ docker_network.py
│     │  ├─ host_network.py
│     │  └─ permissions.py
│     ├─ browsers/              # Chrome、Edge、Firefox 驱动差异
│     │  ├─ chrome.py
│     │  ├─ edge.py
│     │  └─ firefox.py
│     ├─ actions/               # 容器内执行逻辑
│     │  ├─ browser_capture.py
│     │  └─ url_collection.py
│     ├─ ingestors/             # CSV、多 CSV、数据库和代理采集编排
│     │  ├─ csv.py
│     │  ├─ multi_csv.py
│     │  ├─ database.py
│     │  └─ clash.py
│     └─ cli/
│        └─ main.py
├─ profiles/                    # 浏览器、代理和网络模式配置
│  ├─ chrome.yaml
│  ├─ edge.yaml
│  ├─ firefox.yaml
│  ├─ firefox_ech.yaml
│  └─ clash_firefox.yaml
├─ scripts/
│  ├─ capture/                  # 稳定命令包装器
│  ├─ datasets/                 # CSV 转换、合并、裁剪和去重
│  └─ maintenance/              # PCAP 检查、清理、迁移和解密
├─ data/
│  ├─ examples/                 # 可提交的小型测试数据
│  └─ input/                    # 本地输入，默认不提交
├─ runtime/                     # 全部加入 .gitignore
│  ├─ workspaces/
│  ├─ logs/
│  ├─ results/
│  └─ tmp/
├─ vendor/                      # 确实需要随项目管理的第三方内容
├─ docs/
│  ├─ architecture/
│  ├─ operations/
│  └─ archive/
├─ tests/
├─ pyproject.toml
├─ README.md
└─ TASK_QUEUE.md
```

该结构是目标方向，不要求在单次改动中全部建立。实际目录应在入口清单和依赖关系确认后逐步落地。

## 5. 现有目录迁移建议

| 现有内容 | 建议位置 | 说明 |
|---|---|---|
| `tools/chrome.py`、`edge.py`、`firefox.py` | `src/traffic_ingestor/browsers/` | 保留浏览器差异，不保存任务配置 |
| `tools/base_action.py`、`capture.py` | `src/traffic_ingestor/actions/` | 作为容器内公共执行逻辑 |
| `trafficIngestor/base_traffic_ingestor.py` | `core/` 与 `infrastructure/` | 按职责拆分，不直接整体移动 |
| `trafficIngestor/traffic_capture_*.py` | `scripts/capture/` 或统一 CLI | 旧文件先保留为兼容入口 |
| `trafficIngestor_clash/` | `ingestors/clash.py` 与 profiles | Clash 是运行模式，不应形成第二套调度树 |
| `multi_csv_traffic_ingestor/` | `ingestors/multi_csv.py` | 复用公共调度和产物模块 |
| `traffic_capture_single_csv*/action.py` | `actions/` 与 profiles | 避免按组合维度复制目录 |
| `small_tools/code/` | `scripts/datasets/` | 数据加工脚本按用途命名 |
| PCAP 检查、裁剪、解密脚本 | `scripts/maintenance/` | 与普通数据转换脚本分开 |
| `document/`、`monthly/` | `docs/` | 分为架构、运维和归档资料 |
| `temp/` | `runtime/tmp/` 或 `docs/archive/` | 逐项判断，不整体保留 |
| `config/` | `profiles/` 或 `config/templates/` | 区分运行 profile 与第三方模板 |
| `clash-for-linux/` | `vendor/` 或外部构建依赖 | 明确版本、来源和更新方式 |

## 6. 关键设计调整

### 6.1 用运行配置替代复制入口

建议最终提供统一命令：

```powershell
python -m traffic_ingestor capture --profile chrome --csv small_tools/result/tasks.csv
python -m traffic_ingestor capture --profile firefox-ech --csv small_tools/result/tasks.csv
python -m traffic_ingestor capture --profile firefox --proxy clash --csv small_tools/result/tasks.csv
```

下列维度应能够独立配置：

- 浏览器与 Docker 镜像。
- 是否启用 ECH、ML-KEM 等浏览器能力。
- 普通网络、固定 IP 或 Clash。
- CSV、CSV 目录或数据库任务源。
- 并发数、重试次数和冷却时间。
- 结果目录和产物保留策略。

数据集名称不应再进入核心 Python 类名。Bsky、Mastodon、Threads 等入口通常只需要指定不同 CSV 和输出目录。

### 6.2 拆分 `BaseTrafficIngestor`

建议逐步抽取以下组件：

- `DockerContainerManager`：容器生命周期和 `docker exec`。
- `DockerNetworkManager`：网络创建、子网、网关、固定 IP 和冲突诊断。
- `HostNetworkManager`：docker0、bridge 和 veth offload 设置。
- `CsvTaskRepository`：CSV 读取、字段校验和原子删除记录。
- `ArtifactStore`：产物路径计算、搬运、权限和命名。
- `CompletenessVerifier`：基于任务清单检查产物完整性。
- `TaskScheduler`：队列、worker、重试、节流和动态容器数。
- `ExecutionLogger`：任务生命周期日志和进度统计。

`TrafficIngestor` 最终只负责装配组件和控制整体执行流程，不直接包含所有底层实现。

### 6.3 建立统一运行工作区

建议每次运行使用独立目录：

```text
runtime/workspaces/<run_name>/
├─ action.py
├─ meta/
├─ pcap/
├─ html/
├─ ssl_key/
├─ screenshots/
└─ diagnostics/
```

`run_name` 应由入口或命令参数明确生成，并经过字符和长度校验。源码目录不再接收任务运行期间的动态写入。

## 7. 分阶段实施计划

### 阶段 0：建立迁移基线

- 生成现有入口清单，记录浏览器、镜像、代理、网络、CSV、action 和输出目录。
- 标记入口状态：生产使用、实验使用、历史停用、无法确认。
- 保存当前语法检查命令和最小烟测命令。
- 检查远程脚本、tmux 命令或外部计划任务是否依赖现有路径。

验收条件：每个准备迁移的入口都有明确所有者、用途和验证方法。

### 阶段 1：整理非核心内容

- 建立 `docs/`、`scripts/` 和 `runtime/`。
- 将文档、数据处理脚本和维护脚本分类。
- 修正 `single_traffice_capture`、`batch_traffice_capture` 等拼写，但保留临时兼容入口。
- 清理或归档 `temp/` 中可确认的历史内容。
- 明确 `clash-for-linux` 的依赖管理方式。

验收条件：顶层目录能够直接区分源码、脚本、文档、配置、数据和运行状态。

### 阶段 2：建立标准 Python 包

- 新建 `src/traffic_ingestor/` 和 `pyproject.toml`。
- 统一包名为小写 `traffic_ingestor`。
- 旧入口仅导入新包并调用对应 `main()`，暂不删除。
- 消除入口脚本中重复的 `sys.path.insert`。

验收条件：旧命令仍可运行，新模块可以通过标准 Python 包方式导入。

### 阶段 3：收敛 CSV 采集入口

- 抽取公共 `CsvTrafficIngestor`。
- 统一 `fetch_jobs`、成功后删除 CSV 和单轮执行控制。
- 将站点差异改为配置，停止新增站点专用类。
- 保留 Firefox、Edge 等真正有浏览器差异的 profile。

验收条件：新增一个普通 CSV 数据集不需要再复制 Python 文件或 action 目录。

### 阶段 4：拆分基础设施职责

- 优先抽取 CSV、产物和完整性校验，它们对 Docker 运行环境依赖较少。
- 再抽取 Docker 容器和网络管理。
- 最后抽取 worker 调度和生命周期控制。
- 每次只迁移一个职责，避免同时改变路径和行为。

验收条件：主编排类保持较短，组件职责单一，原有采集行为和产物结构不变。

### 阶段 5：统一 CLI 和 profile

- 提供 `capture`、`collect-urls`、`check-dataset` 等子命令。
- 将浏览器、代理和网络模式改为可组合 profile。
- 对路径、镜像和并发配置做启动前校验。
- 配置文件不保存密钥、节点密码或生产凭证。

验收条件：主要生产任务都可通过统一 CLI 表达，不再依赖复制入口。

### 阶段 6：删除兼容层和历史目录

- 搜索仓库、远程运维文档和计划任务中的旧路径引用。
- 对旧入口输出弃用提示，并保留至少一个迁移周期。
- 完成真实单站点和批量采集验证后再删除旧目录。
- 同步更新 README、AGENTS.md、运维文档和任务队列。

验收条件：没有已知调用方依赖旧路径，旧目录删除后验证命令全部通过。

## 8. 推荐的第一批改动

第一批应以低风险整理和建立证据为主，不修改 Docker 调度核心：

1. 编写入口与运行配置映射表。
2. 新建 `docs/architecture/`、`scripts/datasets/`、`scripts/maintenance/`。
3. 分类整理 `document/`、`monthly/` 和 `small_tools/code/`。
4. 将 `temp/` 内容标记为删除、归档或运行时文件。
5. 设计统一 `runtime/` 布局，但暂不切换生产路径。
6. 修复 `TASK_QUEUE.md` 的历史编码损坏记录。
7. 增加覆盖现有入口的批量 `py_compile` 检查脚本。

完成上述工作后，再进入包结构和核心代码拆分阶段。

## 9. 验证要求

每个迁移批次至少执行：

```powershell
python -m py_compile <本次修改的 Python 文件>
```

涉及导入路径时，还应验证：

```powershell
python -c "import traffic_ingestor"
```

涉及调度、挂载或运行工作区时，应在具备 Docker、浏览器驱动和 tcpdump 的环境执行一次缩小版 CSV 单站点采集，并检查：

- 容器是否使用预期镜像、网络和挂载目录。
- PCAP、TLS keylog、HTML 和截图路径是否保持正确。
- 成功任务是否正确从 CSV 移除。
- 失败任务是否保留足够诊断证据。
- 任务结束后是否只清理当前运行命名空间。

## 10. 风险与控制措施

### 路径兼容风险

Docker 挂载、远程命令和外部脚本可能硬编码旧目录。迁移前必须使用仓库搜索和远程运行清单确认引用，旧入口应暂时保留。

### 行为变化风险

移动代码时不应同时调整重试、并发、等待和清理策略。目录迁移与行为优化应拆成不同任务和提交。

### 产物丢失风险

切换运行工作区前应确认产物搬运完成后才进行清理，并对删除范围使用解析后的绝对路径校验。

### 配置泄露风险

Clash 节点、密码、密钥和远程服务器信息不得进入 profile。profile 只保存非敏感结构，敏感值继续通过本地忽略文件或环境变量提供。

### 第三方依赖风险

`clash-for-linux` 应明确采用固定版本的 vendor、Git 子模块或可重复构建脚本之一，不能继续处于来源和更新方法不明确的状态。

## 11. 不建议的做法

- 不建议一次性重命名全部目录并同步修改所有路径。
- 不建议只移动 `base_traffic_ingestor.py` 而不拆分职责。
- 不建议在新目录中复制一套旧实现以维持兼容。
- 不建议继续为每个数据集创建一个新的采集器类。
- 不建议将真实输入数据、日志、抓包和运行工作区提交到 Git。
- 不建议在目录重构期间顺带修改采集成功判定或安全清理逻辑。

## 12. 最终完成标准

目录结构优化完成后，应达到以下状态：

- 仓库顶层只保留少量、职责明确的目录。
- 源码、配置、数据、文档、第三方依赖和运行状态完全分离。
- 浏览器、代理、网络和任务源可以通过配置组合。
- 新增普通采集任务不需要复制入口脚本和 action 目录。
- `BaseTrafficIngestor` 的主要职责已拆入独立组件。
- 旧入口经过兼容周期后安全删除。
- README 和运维文档只描述统一入口及当前有效结构。
- 直接相关的语法检查和最小 Docker 烟测均有可重复记录。
