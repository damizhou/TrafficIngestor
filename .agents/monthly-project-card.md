# 月度项目卡生成流程

最后更新：2026-07-05 09:51:00

## 用途

本文件是可复用的 Codex 工作流提示词，用于在任意项目中生成统一格式的月度项目卡。月度项目卡用于后续跨项目汇总和整月 PPT 制作，不是 10 页 PPT 提纲，也不直接生成 `.pptx`。

## 在其他项目中的使用方式

在目标项目的 Codex 会话中直接输入：

```text
请读取 /home/pcz/code/paper_reader/.agents/prompts/monthly-project-card.md，
按照其中要求，为本项目生成 2026-06 的月度项目卡。
输出到 monthly/2026-06.md。
```

如果目标项目不允许写文件，则输入：

```text
请读取 /home/pcz/code/paper_reader/.agents/prompts/monthly-project-card.md，
按照其中要求，为本项目生成 2026-06 的月度项目卡。
只在对话中输出，不修改文件。
```

如果一次处理多个项目，则输入：

```text
请读取 /home/pcz/code/paper_reader/.agents/prompts/monthly-project-card.md，
分别为以下项目生成 2026-06 月度项目卡，并最后生成总表：
1. /path/to/project_a
2. /path/to/project_b
3. /path/to/project_c
```

## 必要输入

用户至少应提供：

- 目标月份，例如 `2026-06`。
- 目标项目路径，或明确说明“当前目录就是目标项目”。
- 输出方式：写入项目文件，或只在对话输出。

如果缺少目标月份或目标项目路径，先用一句话追问。不要猜月份，也不要跨项目扫描用户未指定的目录。

## 工作原则

- 先读取目标项目的 `AGENTS.md`、`README.md`、`TASK_QUEUE.md` 或等价项目说明；若不存在，说明缺失并继续用可发现材料整理。
- 修改文件前先运行或等价检查 `git status --short --untracked-files=all`，确认工作区已有改动，不覆盖用户未提交内容。
- 只整理目标月份内的工作，不把目标月份之外的后续结果写成本月完成。
- 没有证据的内容不能写成完成态；只能写成“待核验”或“风险”。
- 证据优先级为：验收报告 / 输出汇总 / 测试结果 / commit / 任务队列 / 文档更新时间 / 用户明确说明。
- 不输出密钥、令牌、完整凭证、真实敏感数据、大型原始数据路径明细或隐私内容。
- 如果目标项目是代码项目，优先引用测试、构建、日志摘要和产物路径；如果是文档项目，优先引用任务队列、方案文档、验收文档和最终稿路径。

## 推荐信息收集

在目标项目内按需使用以下命令或等价方式：

```bash
git status --short --untracked-files=all
git log --since="<MONTH_START>" --until="<NEXT_MONTH_START>" --oneline --stat
rg -n "<YYYY-MM>|已完成|验收|最后更新|TODO|风险|结果|output|report" .
find . -type f -newermt <MONTH_START> ! -newermt <NEXT_MONTH_START>
```

月份应根据用户指定值替换。不要在不了解项目规则时运行会刷新产物、删除文件、迁移数据或联网下载依赖的命令。

## 输出文件路径

默认输出路径：

```text
monthly/<YYYY-MM>.md
```

如果目标项目已有月报、周报、任务队列或文档规范，应优先遵循目标项目规则；否则创建 `monthly/` 目录并写入对应月份文件。

## 输出模板

生成的月度项目卡必须使用以下结构。字段可以写“本月未形成……”，但不能删除固定章节。

````markdown
# 项目名：<project-name>

最后更新：YYYY-MM-DD HH:mm:ss

时间范围：YYYY-MM-01 至 YYYY-MM-DD
项目路径：<path>
项目定位：<一句话说明项目做什么>

## 1. 本月目标

- <目标 1>
- <目标 2>

## 2. 本月完成

- 已完成并验证：<完成项 1>  
  证据：<文件路径 / commit / 输出报告 / 测试结果>
- 已完成并验证：<完成项 2>  
  证据：<文件路径 / commit / 输出报告 / 测试结果>
- 待核验：<没有充分证据但可能完成的事项>  
  当前证据：<已有线索或“暂无”>

## 3. 关键产物

| 产物 | 路径 / 证据 | 用途 |
|---|---|---|
| <产物名> | `<path-or-commit>` | <用途> |

## 4. 可量化结果

| 指标 | 数值 | 来源 |
|---|---:|---|
| <指标名> | <数值> | `<path-or-report>` |

若本月没有可量化结果，写：

本月未形成可量化结果。

## 5. 当前风险

- <风险 1>：<影响和建议处理方式>
- <风险 2>：<影响和建议处理方式>

若没有明确风险，写：

本月未发现需要单独上升到月度汇报的风险；仍需以项目测试和验收结果为准。

## 6. 7 月计划

- <下一步 1>
- <下一步 2>

## 7. 汇总口径

一句话总结：<本项目 6 月最重要的结果>

可放入整月总 PPT 的表达：

```text
<1-2 句可直接放入汇总 PPT 的表达>
```
````

## 内容压缩规则

- 一个项目月度卡建议控制在 1-3 页 Markdown，不写成详细日报。
- “本月完成”最多保留 5-8 条，按影响大小排序。
- “关键产物”优先放最终产物，不放临时草稿和中间缓存。
- “可量化结果”只写有来源的数字，不为凑表格编指标。
- “当前风险”只写会影响交付、投稿、实验结论、运行稳定性或后续计划的风险。
- “7 月计划”必须能执行、能验证，不写泛泛目标。

## 验收检查

生成后必须检查：

- 是否包含固定 7 个章节。
- 时间范围是否和用户指定月份一致。
- 每个“已完成并验证”是否有证据。
- 是否把无证据内容降级为“待核验”或“风险”。
- 是否没有混入目标月份之外的后续成果。
- 如果写入文件，是否更新了该文件顶部 `最后更新`。
- 如果目标项目要求维护任务队列，是否按其规则登记和完成任务。
- 运行 `git diff --check` 或等价检查，确认没有 Markdown 空白错误。

## 汇总到整月 PPT 的接口

所有项目都生成月度项目卡后，再生成总表：

```markdown
| 项目 | 当月主线 | 关键完成 | 代表产物 | 当前状态 | 下月动作 |
|---|---|---|---|---|---|
| <项目名> | <主线> | <完成> | <产物> | <状态> | <动作> |
```

最终 PPT 只从这些月度项目卡和总表抽内容，不直接从散乱原始材料抽内容。这样可以保证跨项目格式一致、证据可追溯、风险口径统一。
