# 项目重复功能脚本检查报告

最后更新：2026-07-15 15:21:02

## 1. 结论

本次检查确认：

- 未发现字节内容完全相同的脚本。
- 发现 6 组“结构完全相同、仅名称或常量不同”的有效脚本对。
- 检查时最主要的重复来自 22 个单 CSV 宿主入口；该项已于 2026-07-15 完成配置驱动改造并删除全部旧入口，每组仅保留一个统一 profile CLI。
- Firefox 的 ML-KEM 启用/禁用实验形成了宿主入口、容器 Action、浏览器驱动三层复制，其中驱动层约 460 行代码仅有少量配置差异，是当前风险最高的重复。
- 另有 3 组可以直接合并的小脚本，以及 5 类“功能不同但基础设施重复”的脚本。

建议优先处理顺序：

1. 合并 Firefox 正常/禁用分支。
2. 将 22 个单 CSV 宿主入口改为配置驱动。
3. 删除或合并无引用的旧入口和小工具副本。
4. 最后抽取 Docker 容器池、CSV 原子写入等跨模块公共能力。

## 2. 检查范围与方法

检查范围覆盖项目工作区中的 Python、Shell 和 JavaScript 脚本：

- 文件系统扫描到 117 个脚本候选：89 个 Python、6 个 Shell、22 个 JavaScript。
- 其中 JavaScript 全部位于 clash-for-linux/dashboard/public/，属于构建后的第三方 Dashboard 静态资源；对其做了文件哈希检查，但不作为本仓库可维护业务脚本进行语义合并。
- .git/、.venv/、IDE 缓存和运行产物不进入语义检查。
- 本地忽略文件 config/sever_info.py 只按结构纳入扫描，未在报告中展开其配置内容。

使用的判断依据：

1. SHA-256 检查字节级副本。
2. 去除注释、空白并泛化标识符和常量后的结构指纹。
3. Python 函数和类方法的 AST 指纹对比。
4. 对高相似度文件逐组阅读差异，确认是否只是参数、路径、浏览器或运行方式不同。
5. 搜索仓库引用，区分仍在使用的变体、无引用旧脚本和第三方生成文件。

“功能重复”分为两类：

- 直接重复：可以由同一实现加参数完成，适合合并或删除副本。
- 部分重复：用户功能不同，但大量容器管理、文件处理或进程管理代码重复，适合抽公共层，不应直接删除任一脚本。

### D2. Firefox ML-KEM 启用/禁用形成三层重复

严重度：高

涉及文件：

- 宿主入口：
  - trafficIngestor/traffic_capture_single_csv_firefox.py
  - trafficIngestor/traffic_capture_single_csv_firefox_disable.py
- 容器 Action：
  - traffic_capture_single_csv_firefox/action.py
  - traffic_capture_single_csv_firefox_disable/action.py
- 浏览器驱动：
  - tools/firefox.py
  - tools/firefox_disable.py

重复证据：

- 两个宿主入口均为 73 行，除 HOST_CODE_PATH、输出路径和 CSV 路径外逻辑相同。
- 两个 Action 均为 49 行，实际差异只有导入 tools.firefox 还是 tools.firefox_disable。
- 两个驱动分别约 466/460 行，主体几乎相同；核心行为差异是：
  - security.tls.enable_kyber 为 True 或 False。
  - network.http.http3.enable_kyber 为 True 或 False。
  - 正常驱动额外支持 preference_overrides。
- tools/firefox_ech.py 已通过 preference_overrides 调用公共 Firefox 驱动，证明同一模式可以覆盖实验开关，无需复制完整驱动。

建议：

- 删除 tools/firefox_disable.py 的完整副本，改为一个很薄的工厂函数，向 tools.firefox.create_firefox_driver() 传入两个禁用 preference。
- Firefox Action 通过类属性或环境变量选择 preference 集，不再复制 Action 文件。
- 宿主入口归入 D1 的配置驱动方案，用 firefox_profile=default|disable_ml_kem|force_ech 表达实验分支。
- 合并前必须分别验证默认、禁用 ML-KEM、强制 ECH 三种浏览器启动参数和 keylog 结果。

## 4. 部分重复：应抽公共层，但不能直接删除脚本

### P1. 新闻接收器单任务与批任务宿主脚本重复 Docker 调度框架

涉及文件：

- trafficIngestor/news_receiver_traffic_ingestor.py
- trafficIngestor/news_receiver_traffic_batch.py

共同部分包括用户名解析、宿主目录清理、命令执行、Docker 可用性检查、容器存在/运行检查、创建和启动容器、关闭网卡 offload、容器名生成、权限修正、worker、信号处理和容器池准备。

两者的任务协议不同：一个按单 URL 处理，另一个按同域名 URL 批次处理，因此不应删除任一文件。建议提取共同的旧版容器池基类，或迁移到 BaseTrafficIngestor 的现有能力。

### P2. get_url_list.py 与 BaseTrafficIngestor 重复容器池管理

涉及文件：

- trafficIngestor/get_url_list.py

建议把与业务无关的 Docker 容器池能力下沉为组合式组件，例如 DockerContainerPool，由两条业务链复用。

### P3. BatchAction 与 BaseAction 重复进程和抓包生命周期

涉及文件：

- tools/action_batch.py
- tools/base_action.py

重复内容包括僵尸进程收割、启动/停止抓包、创建/关闭浏览器、清理残留进程、结果 JSON 和命令行 JSON 入口。批模式的“多个 URL 共用一个 pcap/keylog”是独立语义，应保留。

建议让二者复用较小的进程生命周期和结果写入组件，不建议强行让批模式完全套用单 URL 的 BaseAction.start_task()。

### P4. 两个 CSV 重写工具重复备份与原子替换框架

涉及文件：

- small_tools/code/adjust_csv_repeat_count.py

两者业务分别是补齐重复次数和限制重复次数，但共享路径解析、备份文件命名、权限保留、临时文件写入和替换流程。建议抽取 small_tools/code/csv_file_ops.py，不要合并两个用户命令。

### P5. URL 收集子项目重复浏览器和日志工具

涉及文件：

- url_list_collector/utils/chrome.py 与 tools/chrome.py
- url_list_collector/utils/logger.py 与 tools/logger.py

URL 收集子项目需要更轻量的 Scrapy/Selenium 配置，直接依赖完整 tools/chrome.py 可能引入不必要耦合。建议只抽取稳定的小函数，例如容器日志路径、进程清理和基础 Chrome option 构造；不建议直接用一个巨大模块替换另一个。

## 5. 高相似但不应判定为重复脚本的项目

- small_tools/code/merge_csv.py 与两个递归合并工具：前者处理指定主 CSV/临时 CSV、备份和业务去重；后两者只是目录递归拼接，不能直接互相替代。
- small_tools/code/convert_github_repos_to_csv.py、convert_social_url_lists_to_csv.py、url_txt_to_csv.py：输出都为 CSV，但输入格式、字段映射和域名生成规则不同。
- small_tools/code/check_pcap_dataset.py 与 small_tools/code/trim_pcap_by_id_domain.py：前者检查数据集完整度并可清理不合格域名，后者按 id + domain 限量并同步清理旁路产物。
- tools/chrome.py、tools/edge.py、tools/firefox.py：浏览器驱动目标相似，但实现差异足够大；应只抽取 _host_matches_domain、host resolver 等小型通用函数。
- multi_csv_traffic_ingestor/traffic_capture_csv_directory.py：虽然覆盖了部分基类钩子，但其多 CSV、公平轮转和每站目标数量语义独立，不属于单 CSV 入口副本。
- 两个空的 __init__.py 被结构指纹判为相同，但没有可合并的运行功能。
- clash-for-linux/dashboard/public/assets/*.js 是压缩构建产物，不应在本仓库手工合并。

## 7. 总体风险

- 各 profile 仍包含硬编码远程目录、镜像、CSV 和等待时间；后续修改配置时必须继续逐项核对运行环境。
- 配置驱动入口允许 HOST_CODE_PATH 在仓库中不存在；运行前会创建独立工作目录并同步公共 action，这属于当前设计而不是入口失效证据。
- 项目没有正式 pytest 体系，合并采集入口后至少应运行 py_compile，并对 Chrome、Edge、Firefox、Clash、ML-KEM 禁用和 ECH 各做一个最小单站点烟测。
- D1 与 D3 已完成配置驱动整改；其余重复项仍按本报告建议分批处理。
